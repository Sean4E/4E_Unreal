Foundation
Understanding Blueprints

Before you wire anything, understand what Blueprints are, how the visual scripting system works, and why it is the fastest way to prototype gameplay in UE5.

00

What Are Blueprints?

Blueprints are UE5's visual scripting system. Instead of typing lines of code, you drag colourful nodes onto a graph and connect them with wires. Each node performs a single action — play a sound, move an object, check a condition — and the wires define the order things happen. Every gameplay feature in Unreal Engine 5 can be built entirely with Blueprints.

Think of Blueprints like a flowchart. Each box is an instruction, and the arrows between boxes say "do this, then do that." The engine reads your flowchart from left to right and follows every connection.

Visual Scripting vs Traditional Code

Traditional Code (C++)Blueprints (Visual Scripting)
Write text in a code editorDrag nodes onto a visual graph
Syntax errors if you mistypeWires only connect compatible pins
Must compile from command lineOne-click Compile button
Harder to learn, faster at runtimeEasier to learn, fast enough for most gameplay
Actor->SetActorRotation(Rot);Node: Set Actor Rotation with a wire
No code needed. Blueprints compile down to the same engine functions as C++. For gameplay logic, Blueprints are the standard workflow used by professional studios.

The Event Graph: Where Logic Lives

Every Blueprint Actor has an Event Graph — a dark canvas with a grid background. This is your workspace. You place nodes here and connect them with wires to create behaviour.

  • Execution flow — follows the white wires from left to right. White wires say "do this, then do that."
  • Data flow — coloured wires carry values (numbers, text, true/false, positions). Data wires feed information into nodes.

The Three Pillars

  • Events — the starting points of your logic. Something happens (the game starts, the player presses a key, an actor overlaps a trigger) and an event node fires.
  • Functions — reusable groups of nodes that perform a task. Call them from anywhere in the same Blueprint. Think of them as named recipes you can reuse.
  • Variables — named containers that store values (a number, a true/false, a colour, a reference to another object). Your Blueprint can read and write these at any time.

Your First Blueprint Actor

Time to create your first reusable Blueprint. You will build BP_HelloCube — an Actor that contains a glowing cube with a warm point light. This is the Blueprint you will add behaviour to throughout the entire masterclass.

  1. In the Content Browser, right-click in an empty area
  2. Select Blueprint Class
  3. Choose Actor as the parent class
  4. Name it BP_HelloCube
  5. Double-click to open the Blueprint Editor

Add Components

Components are the building blocks of any Actor. They define what the Actor looks like, sounds like, and how it interacts with the world.

  1. In the Components panel (top-left), click + Add
  2. Search for Static Mesh and add it — this gives your Actor a visible shape
  3. In the Details panel (right side), find Static Mesh and set it to Cube (search "Cube" in the mesh dropdown)
  4. Click + Add again
  5. Search for Point Light and add it
  6. Select the Point Light in the Components panel
  7. In the Details panel, set Light Color to a warm orange
  8. Set Intensity to 5000
  9. In the Viewport, drag the light slightly above the cube so it illuminates the top face

Understanding the Component Hierarchy

Every Actor has a DefaultSceneRoot at the top. This is the invisible "anchor point" of the Actor in the world. All other components are children of this root.

  • DefaultSceneRoot — the Actor's position in the world. Moving this moves everything.
  • StaticMesh — the visible cube mesh. A child of the root, so it moves with the Actor.
  • PointLight — the light source. Also a child of the root.
The hierarchy is like a skeleton. The root is the spine. Attach a mesh (the body) and a light (a glowing accessory) to the spine, and they all move together as one unit.

The Event Graph — Making Things Happen

The Event Graph is where your Blueprint comes alive. Open BP_HelloCube and click the Event Graph tab at the top of the Blueprint Editor. You will see a dark canvas with a grid — this is where you place nodes and connect them.

Event BeginPlay: Print a Message

Event BeginPlay fires once when the game starts (or when this Actor is spawned into the world). It is the most common starting point for setup logic.

  1. You should see Event BeginPlay already on the graph (red header node)
  2. Drag from the white execution pin on the right side of BeginPlay into empty space
  3. A search menu appears — type Print String
  4. Click Print String to place the node
  5. In the Print String node, change the text to Hello from my Blueprint!
  6. Compile and Save
  7. Back in the main editor, press Alt + P to Play in Editor
  8. Look at the top-left of the Viewport — you should see "Hello from my Blueprint!"

Event Tick: Make the Cube Spin

Event Tick fires every single frame (60+ times per second). Use it for continuous actions like rotation and movement — but use it sparingly because it runs constantly.

  1. Find Event Tick on your graph (it may already be there, or right-click and search "Event Tick")
  2. Drag from Event Tick's exec pin into empty space
  3. Search for AddActorLocalRotation and place it
  4. On the Delta Rotation pin, click the expand arrow to reveal X, Y, Z
  5. Set Y (Pitch) to 1.0 — this rotates 1 degree per frame around the Y axis
  6. Compile, Save, and Play
  7. Your cube should now be spinning continuously!
Performance warning: Never put heavy logic on Event Tick. If you only need to react to something happening, use events and overlaps instead. Tick should be a last resort for continuous movement.

03

Variables — Storing Data

A variable is a named container for data. Instead of typing the number 1.0 directly into the rotation node, you store it in a variable called RotationSpeed. Now you can change the speed from one place, and every node that reads it gets the new value.

A variable is like a labelled jar. The label says what is inside ("RotationSpeed"), and you can change the contents (1.0, 2.5, 10.0) at any time without replacing the jar.

Create Your Variables

In the My Blueprint panel (bottom-left of the Blueprint Editor), find the Variables section.

  1. Click the + button next to Variables
  2. Name it RotationSpeed
  3. In the Details panel on the right, set the type to Float (light green pin)
  4. Click Compile, then set the Default Value to 1.0
  5. Create another variable: click + again
  6. Name it bShouldSpin
  7. Set the type to Boolean (dark red pin)
  8. Compile, then set Default Value to true (checked)
Naming convention: Booleans are prefixed with a lowercase b in Unreal — bShouldSpin, bIsOpen, bHasKey. This is an Epic convention you will see everywhere.

The Eye Icon: Making Variables Public

Next to each variable in the My Blueprint panel, there is a small eye icon. Click it to make the variable Instance Editable (public). This means:

  • You can change the variable's value in the Details panel of each instance in the level
  • Each copy of BP_HelloCube can have a different RotationSpeed
  • The eye turns open when the variable is public

Click the eye icon on RotationSpeed to make it editable. Now when you select a BP_HelloCube in the level, you will see RotationSpeed in its Details panel and can set a unique value per instance.

04

Branch & Flow Control

Flow control nodes let your Blueprint make decisions. The most important one is Branch — it is the Blueprint equivalent of an if/else statement. Give it a Boolean condition, and it sends execution down either the True or False path.

Branch: Use bShouldSpin to Control Rotation

Right now your cube spins on every Tick unconditionally. Let's add a condition so it only spins when bShouldSpin is true.

  1. Disconnect the wire between Event Tick and AddActorLocalRotation (Alt+click the wire)
  2. Drag from Event Tick's exec pin into empty space → search Branch
  3. Drag bShouldSpin from Variables onto the graph (Get node)
  4. Connect the Boolean output to the Branch's Condition pin (dark red)
  5. Connect Branch's True exec pin to AddActorLocalRotation's exec pin
  6. Leave the False pin disconnected (nothing happens when bShouldSpin is false)
  7. Compile, Save, Play — the cube spins only when bShouldSpin is true

Let's use the PointLight on BP_HelloCube to practice FlipFlop:

Input — Responding to the Player

UE5 uses the Enhanced Input System to handle keyboard, mouse, and gamepad input. Instead of directly listening for a key press, you create Input Action assets and map keys to them through an Input Mapping Context. This makes it easy to rebind keys later.

Set Up Enhanced Input (One-Time Setup)

  1. In the Content Browser, right-click → Input → Input Action
  2. Name it IA_Interact
  3. Double-click to open it — leave Value Type as Digital (bool) and close
  4. Right-click again → Input → Input Mapping Context
  5. Name it IMC_Default
  6. Double-click to open it
  7. Click + Add Mapping, select IA_Interact
  8. Under the mapping, click + to add a key and set it to E
  9. Save everything
☰ IMC_Default — Input Mapping Context
Mappings
IA_InteractE

Add the Mapping Context to Your Player

The mapping context must be added to the player controller so the engine knows to listen for these inputs. If you are using the Third Person Template, this is likely already set up. If not:

  1. Open your player character Blueprint (e.g. BP_ThirdPersonCharacter)
  2. In Event BeginPlay, add an Add Mapping Context node
  3. Set the Mapping Context to IMC_Default
  4. Wire it from BeginPlay's exec pin
Template shortcut: The Third Person and First Person templates already include Enhanced Input setup. Check your character Blueprint — if Add Mapping Context is already there, you just need to make sure your IMC_Default is included.

Respond to Input in BP_HelloCube

Now let's make pressing E toggle the cube's spinning on and off:

  1. Open BP_HelloCube's Event Graph
  2. Right-click → search IA_Interact (your Input Action)
  3. Place the IA_Interact event node — it has Started, Triggered, and Completed pins
  4. Drag from Started (fires once when the key is pressed) into empty space
  5. Add a Flip Flop node
  6. From FlipFlop A: add Set bShouldSpin → value: true
  7. From FlipFlop B: add Set bShouldSpin → value: false
  8. Compile, Save, Play — press E to start/stop the cube spinning!
Important: For the Actor to receive input, you must select BP_HelloCube in the level, go to its Details panel, and under Input → Auto Receive Input, set it to Player 0. Without this, the Actor ignores keyboard events.

Collision & Overlap Events

Overlap events let your Blueprint detect when another actor enters or leaves a zone. This is how you build proximity triggers: walk near a door and it opens, step on a pressure plate and a trap fires, enter a room and the lights turn on.

Add a Box Collision to BP_HelloCube

  1. Open BP_HelloCube and go to the Components tab (Viewport)
  2. Click + Add → search Box Collision
  3. Select the Box Collision in the Components panel
  4. In the Details panel, set Box Extent to X: 200 Y: 200 Z: 200 — this creates a large invisible trigger zone around the cube
  5. Under Collision, set Collision Presets to OverlapAllDynamic
  6. Make sure Generate Overlap Events is checked (it should be by default)

Overlap Events

UE5 provides several overlap events. The two most common:

EventWhen It FiresUse For
Event ActorBeginOverlapAny actor enters the collision zoneQuick and simple proximity detection
Event ActorEndOverlapAny actor leaves the collision zoneResetting state when the player walks away
OnComponentBeginOverlapSpecific component is overlappedMore precise — when you have multiple collision shapes

The Cast To Pattern

When an overlap fires, you get a reference to the Other Actor that overlapped. But "Other Actor" could be anything — the player, a projectile, a falling box. You need to check if it is the player. This is where Cast To comes in.

Cast To attempts to convert an Actor reference into a more specific type. If it succeeds, you know what you are dealing with and can access that type's variables and functions.

  1. In BP_HelloCube's Event Graph, right-click → Add Event → ActorBeginOverlap
  2. Drag from the Other Actor pin (blue, Object type)
  3. Search for Cast To BP_ThirdPersonCharacter (or your player class)
  4. Connect the exec from ActorBeginOverlap to Cast To's exec
  5. From Cast To's success exec pin, add Set bShouldSpin → true
  6. Add Event ActorEndOverlap
  7. Wire it through another Cast To → on success, Set bShouldSpin → false

Now the cube starts spinning when the player walks near it and stops when the player walks away!

07

Communication Between Blueprints

So far, all your logic lives inside BP_HelloCube. But real projects have many Blueprints that need to talk to each other — a switch that controls a light, a key that opens a door, a button that triggers an elevator. This step covers the three main communication patterns.

Method 1: Direct Reference

The simplest method. Create a variable that holds a reference to a specific Actor in the level.

  1. Create a new Blueprint Actor: BP_Switch
  2. Add a Static Mesh (a small cube or cylinder to represent a button)
  3. Add a Box Collision for the trigger zone
  4. Create a variable: TargetCube with type BP_HelloCube (Object Reference)
  5. Click the eye icon to make it Instance Editable (public)
  6. Place both BP_Switch and BP_HelloCube in the level
  7. Select the BP_Switch instance, and in its Details panel, set TargetCube to the BP_HelloCube instance (use the eyedropper or dropdown)
Limitation: Direct references must be set per-instance in the level. This works well for one-to-one relationships but does not scale to "affect all cubes."

Method 2: Get All Actors of Class

When you want to affect all instances of a Blueprint class, use Get All Actors of Class. It returns an array of every Actor of that type currently in the level.

  1. In BP_Switch's Event Graph, add Event ActorBeginOverlap
  2. Wire through a Cast To to confirm it is the player
  3. After the cast, add Get All Actors of Class → set class to BP_HelloCube
  4. The output is an Array of BP_HelloCube references
  5. Add a For Each Loop to iterate through each one
  6. Inside the loop, use Set bShouldSpin (target: Array Element) to set each cube spinning
Performance note: Get All Actors of Class searches the entire level every time it runs. Do not call it on Tick. Call it once on an event (like an overlap) and cache the result if needed.

Method 3: Casting

You already used Casting in Step 6 with overlaps. Casting is how you safely convert a generic Actor reference into a specific Blueprint type so you can access its unique variables and functions.

Event Dispatchers (Preview for Part 2)

Event Dispatchers are the most powerful communication method. They let a Blueprint broadcast an event that other Blueprints can subscribe to. Think of it like a radio station: one Blueprint transmits, and any number of others can tune in.

  • BP_Switch creates an Event Dispatcher called OnSwitchActivated
  • BP_Switch calls (broadcasts) OnSwitchActivated when the player overlaps
  • BP_HelloCube binds to that dispatcher and starts spinning when it hears the broadcast
