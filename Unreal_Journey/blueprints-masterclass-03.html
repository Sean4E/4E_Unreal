Save & Load System

A SaveGame object is a special UE5 class designed for persistent data. It acts as a container — you define which variables should be saved (health, location, inventory), and the engine serialises them to disk. When the player loads, those variables are deserialised back into memory, restoring the exact game state.

Think of a SaveGame object like a filing cabinet. You decide which folders go inside (your variables), and the engine handles locking and unlocking the cabinet (writing to and reading from disk).

Create the SaveGame Blueprint

  1. In the Content Browser, right-click → Blueprint Class
  2. Under "All Classes", search for SaveGame and select it as parent
  3. Name it BP_MySaveGame
  4. Open it and add variables: PlayerHealth (Float), PlayerLocation (Vector), InventoryItems (Array of Strings)
  5. Compile and Save

Multiple Save Slots

Support multiple saves by varying the Slot Name string. A common pattern is "SaveSlot_1", "SaveSlot_2", "SaveSlot_3". Build a UI that lists available slots, checks which ones exist with Does Save Game Exist, and lets the player choose which to load or overwrite.

Pro tip: Store a "last played" timestamp in each save so you can show "Continue" on the most recent slot.

Practical: Save on Checkpoint, Load on Death

  1. Create an overlap trigger volume at a checkpoint location
  2. On overlap, create a SaveGame object, populate it with current Health and Location
  3. Call Save Game to Slot with slot "Checkpoint"
  4. On player death, call Load Game from Slot with "Checkpoint"
  5. Cast the result to BP_MySaveGame, read PlayerHealth and PlayerLocation
  6. Set the player's health and teleport them to the saved location
Milestone: You can create a SaveGame class, save player data to a named slot, check if saves exist, load them back, and support multiple save slots with async writes for performance.

Inventory System

An inventory system is built on Arrays of Structs. A Struct groups related data together — an item's name, icon, quantity, and type live in one tidy package. An Array holds multiple structs, forming your inventory list. Together, they let you add, remove, stack, and display items in a grid UI.

A Struct is like a card in a card game — it has a name, a picture, and stats all on one card. Your inventory Array is the hand of cards you are holding.

Create the Inventory Struct

  1. Content Browser → right-click → Blueprints → Structure
  2. Name it S_InventoryItem
  3. Add variables: ItemName (String), Icon (Texture2D), Quantity (Integer), ItemType (E_ItemType enum)
  4. Create enum E_ItemType with values: Weapon, Consumable, Material, Key, Quest

Inventory UI Widgets

Build two Widget Blueprints:

  • WBP_InventorySlot — displays one item: icon image, quantity text, border highlight. Receives an S_InventoryItem and updates its visuals.
  • WBP_InventoryGrid — a Uniform Grid Panel or Wrap Box that spawns WBP_InventorySlot widgets for each item in the array.
Drag and drop: Override OnDragDetected in WBP_InventorySlot to create a DragDropOperation. Override OnDrop in the target slot to swap or move items. Use a visual "ghost" widget as the drag visual.

Practical: Full Pickup-to-UI Flow

  1. Create BP_Pickup actor with a StaticMesh, SphereCollision, and an ItemData variable (type: S_InventoryItem)
  2. On overlap with player, call the player's AddItem function, passing ItemData
  3. AddItem checks for stacking, adds to the Inventory array, and calls OnInventoryChanged dispatcher
  4. WBP_InventoryGrid binds to OnInventoryChanged — clears grid, loops through array, spawns slots
  5. Click a slot to call UseItem or DropItem function
Milestone: You can define an inventory struct, add/remove/stack items in an array, display them in a grid widget, and handle pickup-to-UI flow with event dispatchers.

Component Architecture

Actor Components are reusable behaviour modules you attach to any Actor. Instead of duplicating health logic across your player, enemies, and destructible barrels, you write it once in a component and plug it in everywhere. This is composition over inheritance — build complex Actors by combining simple, focused components.

Components are like LEGO bricks. A "Health" brick, a "Movement" brick, a "Combat" brick. Snap different bricks onto different Actors to give them different abilities, without rewriting anything.

Two Types of Components

TypeHas Transform?Use Case
Actor ComponentNo — pure logicHealth system, inventory manager, buff/debuff tracker
Scene ComponentYes — position, rotation, scaleAttach points, weapon sockets, camera booms

Create a Health Component

  1. Content Browser → right-click → Blueprint Class → Actor Component
  2. Name it BP_HealthComponent
  3. Add variables: Health (Float, default 100), MaxHealth (Float, default 100)
  4. Add an Event Dispatcher: OnDeath
  5. Create a function: TakeDamage with input DamageAmount (Float)
  6. Inside TakeDamage: subtract DamageAmount from Health, clamp to 0. If Health <= 0, call OnDeath

Attach to Any Actor

Open any Actor Blueprint (player, enemy, barrel), click + Add Component, search for BP_HealthComponent. Each Actor can have different MaxHealth values — set them per-instance in the Details panel. The component handles all damage and death logic internally.

Communication Patterns

  • Owner reads component: Get Component by Class → call functions directly
  • Component notifies owner: Use Event Dispatchers. The owner binds to OnDeath and reacts (play death animation, destroy actor, etc.)
  • External access: Other Actors get a reference to the target, call Get Component by Class, then call TakeDamage
Milestone: You can create reusable Actor Components, build a HealthComponent with TakeDamage and OnDeath, attach it to multiple Actors, and communicate between components and owners via dispatchers.

Procedural Generation

Procedural generation means creating content at runtime using algorithms instead of hand-placing everything. In Blueprints, this starts with Spawn Actor from Class — a node that creates any Actor at a specified location and rotation while the game is running. Combined with random streams and loops, you can generate entire levels, scatter props, or build dungeons.

Key Concepts

Spline ComponentA curve you define in the editor. Place objects along it at regular intervals for paths, fences, cables, etc.
Instanced Static MeshRenders thousands of identical meshes in one draw call. Use for grass, rocks, scattered props.

Spline-Based Placement

A Spline Component defines a curve in 3D space. You can place objects along it programmatically:

  1. Add a Spline Component to an Actor Blueprint
  2. Shape the spline in the Viewport by adding/moving control points
  3. In BeginPlay, use a For Loop from 0 to GetNumberOfSplinePoints - 1
  4. At each index, call GetLocationAtSplinePoint and GetRotationAtSplinePoint
  5. Spawn or place objects at those transforms
For smooth distribution: Use GetLocationAtDistanceAlongSpline with evenly spaced distances instead of spline points for uniform spacing.

Instanced Static Mesh for Performance

When scattering hundreds of identical meshes (rocks, grass, crates), use an Instanced Static Mesh Component (ISM) instead of spawning individual Actors. ISM renders all instances in a single draw call, drastically reducing overhead.

  • Add an Instanced Static Mesh Component to your generator Actor
  • Set its mesh to your prop (rock, barrel, etc.)
  • In your loop, call Add Instance (World Transform) instead of Spawn Actor
  • For large open worlds, use Hierarchical Instanced Static Mesh (HISM) for automatic LOD and culling

Practical: Dungeon Room Generator

  1. Create 3-4 room Actor Blueprints (BP_Room_A, BP_Room_B, etc.) with doorways at standard positions
  2. Store them in a RoomClasses array (Array of Actor Class)
  3. Create a BP_DungeonGenerator with a RandomStream variable (expose Seed to allow reproducibility)
  4. In BeginPlay, loop N times. Each iteration picks a random room, calculates its position on a grid, and spawns it
  5. Add corridor connectors between adjacent rooms
  6. Scatter props inside each room using ISM with random offsets
Milestone: You can spawn Actors at runtime, use Random Streams for reproducible results, place objects along splines, and use Instanced Static Meshes for efficient prop scattering.

Material Parameter Control

Dynamic Material Instances let you change material properties at runtime from Blueprints. You cannot modify a shared material directly (it would affect every object using it). Instead, you create a Dynamic Material Instance — a per-object copy — and change its parameters freely. This powers health-based colour changes, dissolve effects, damage flashes, and more.

Setup: Parameterised Material

  1. In your Material, create parameters: right-click a value node → Convert to Parameter
  2. Name your parameters clearly: BaseColor, EmissiveStrength, DissolveAmount
  3. Scalar Parameters control numbers (float). Vector Parameters control colours (RGBA)
  4. Apply this material to your mesh in the editor

Dissolve Effect with Timeline

  1. In your material, multiply a Noise Texture by a DissolveAmount scalar parameter
  2. Feed the result into Opacity Mask (set Blend Mode to Masked)
  3. In Blueprint, add a Timeline that lerps DissolveAmount from 0 to 1 over 2 seconds
  4. On each Timeline update, call Set Scalar Parameter Value on DynMaterial
  5. Trigger the Timeline on death: the mesh dissolves away

Material Parameter Collection

Material Parameter Collections (MPC) are global parameter sets. Every material that references an MPC reads the same values. Change one float in the collection, and every material responds.

  • Use case: Day/night cycle — a single "TimeOfDay" float controls sky colour, fog tint, and emissive intensity across all materials
  • Create in Content Browser: right-click → Materials → Material Parameter Collection
  • In Blueprint: Set Scalar Parameter Value and Set Vector Parameter Value on the MPC asset
Performance: MPCs are very efficient. They update a single GPU constant buffer rather than updating each material individually.
Milestone: You can create Dynamic Material Instances, control scalar and vector parameters from Blueprint, build a health-based colour system, create dissolve effects with Timelines, and use Material Parameter Collections for global material control.

Niagara Particle Systems

Niagara is UE5's GPU-accelerated particle system. From Blueprints, you spawn, activate, and control Niagara systems to create hit effects, trails, ambient particles, weather, and much more. You do not need to build Niagara systems from scratch — you can use existing ones and drive them with Blueprint data.

Practical Examples

  • Hit sparks on impact: On the projectile's OnHit event, call Spawn System at Location with the hit point and normal rotation, then destroy the projectile
  • Trail behind projectile: Add a Niagara Component to the projectile Blueprint. Set it to your trail system. It activates automatically when the actor spawns and follows it through the air
  • Environmental ambience: Place a Niagara System actor in the level (dust motes, fireflies, fog wisps). Use Set Niagara Variable to change intensity based on time of day or weather

Lifecycle Management

  • Auto-Destroy: Check "Auto Destroy" on the Niagara Component so one-shot effects clean themselves up after finishing
  • Pooling concept: For frequently spawned effects (bullet impacts), consider an object pool — pre-spawn N Niagara Components, activate/deactivate as needed instead of spawning/destroying each time
  • Deactivate vs Destroy: Deactivate lets particles finish naturally. Destroy kills them immediately. Use Deactivate for graceful fade-out.
Performance: Niagara systems with GPU simulation (GPU Sim) can handle millions of particles. CPU systems are more flexible but slower. Choose based on your needs.
Milestone: You can spawn Niagara effects at locations, attach trail systems to actors, pass Blueprint data into particles, and manage particle lifecycle with auto-destroy and pooling.

Networking Basics (Multiplayer Concepts)

UE5 uses a Server-Client model for multiplayer. One machine is the authoritative Server — it owns the "truth" of the game state. All other machines are Clients that send input to the server and receive updates back. The server validates everything, preventing cheating and ensuring consistency.

Think of the server as a referee. Players (clients) request actions ("I want to open this door"), and the referee decides if it is allowed, then tells everyone the result.

Key Networking Concepts

ConceptWhat It Does
Replicated VariableA variable marked "Replicated" in Details. The server's value is automatically sent to all clients.
RepNotifyA replicated variable that calls a function (OnRep_VariableName) on clients when it changes. Use for visual updates.
Server RPCA function that runs only on the server. Clients call it to request actions.
Client RPCA function the server calls that runs only on the owning client. For client-specific feedback.
Multicast RPCA function the server calls that runs on the server AND all clients. For effects everyone should see.
Has AuthorityReturns true on the server (or for locally controlled actors). Use to gate server-only logic.

RPC Flow: Replicated Door

Here is the complete flow for a door that one player opens and all players see:

  1. Player interacts with door → calls Server_ToggleDoor (Server RPC)
  2. Server receives the call, validates it (is the player close enough?)
  3. Server sets bIsOpen = true (this variable is RepNotify)
  4. Engine replicates bIsOpen to all clients automatically
  5. On each client, OnRep_bIsOpen fires → plays open/close animation
  6. Server also calls Multicast_PlayDoorSound so everyone hears it

Setting Up Replication

  1. In your Actor Blueprint, check Replicates in the Class Defaults
  2. For variables: select the variable, set Replication to Replicated or RepNotify
  3. For functions: select the function, set Replicates to Run on Server, Run on Owning Client, or Multicast
  4. Mark server RPCs as Reliable for important actions (door toggle) or Unreliable for cosmetic ones (particle spawn)
Common mistake: Never change replicated variables on the client. Only the server should set them. Clients read the replicated values and respond visually through RepNotify.
Milestone: You understand the Server-Client model, can mark variables as Replicated/RepNotify, create Server/Client/Multicast RPCs, and build a replicated door that syncs across all players.

Optimisation & Best Practices

Well-organised, performant Blueprints are the mark of a professional developer. This step covers the patterns that separate prototype-quality Blueprints from production-ready ones: avoiding Tick abuse, caching casts, naming conventions, profiling tools, and clean graph organisation.

Avoid Tick Where Possible

Event Tick runs every frame. For 60fps, that is 60 calls per second per Actor. Multiply by hundreds of Actors and you have serious overhead. Use these alternatives:

Instead of Tick...Use ThisWhy
Checking a value every frameEvent DispatchersReact only when value actually changes
Periodic checksSet Timer by EventRuns at your chosen interval, not every frame
Waiting for somethingTimelines / DelaysFire-and-forget, no polling needed
Collision checksOverlap EventsPhysics engine handles detection efficiently
If you must use Tick: Disable it by default (Start with Tick Enabled = false) and only enable it during the exact window you need it. Disable it again when done.

Cache Cast Results

Casting is not free. Every Cast To node performs a type check. If you cast to the same class multiple times (e.g., Cast to BP_PlayerCharacter), cache the result in a variable and reuse it.

Blueprint Nativization

Blueprint Nativization converts Blueprints to C++ during packaging, giving them near-native performance. In UE5, this has evolved into "Blueprint Compilation Manager" optimisations. For critical-path Blueprints (AI, physics, high-frequency logic), consider:

  • Moving hot-path logic to C++ functions and calling them from Blueprint
  • Using BlueprintNativeEvent for functions that need both C++ speed and Blueprint overrideability
  • Keeping Blueprint for prototyping and high-level flow, C++ for low-level performance-critical code

Organisation & Cleanliness

  • Comments: Select a group of nodes → press C to wrap them in a comment box. Describe what the group does.
  • Reroute nodes: Double-click a wire to add a reroute node. Clean up long, tangled wires.
  • Collapse to Function: Select nodes → right-click → Collapse to Function. Reduces visual clutter and makes logic reusable.
  • Collapse to Macro: Same as function but supports multiple exec outputs (latent-friendly).
  • Folders: Organise your Content Browser with clear folder hierarchies: /Blueprints/Characters/, /Blueprints/Items/, etc.

Common Mistakes & Fixes

  • Forgetting to Compile: Always Compile after changes. Uncompiled Blueprints show a yellow question mark and may not work in-game.
  • Circular references: Actor A references Actor B which references Actor A. Use interfaces or dispatchers to break the cycle.
  • Not checking IsValid: Always validate object references before using them. A destroyed actor's reference becomes null.
  • Putting everything in one Blueprint: Split logic into components, functions, and child Blueprints. A 200-node Event Graph is a maintenance nightmare.
  • Ignoring the output log: Warnings and errors appear in the Output Log. Read them — they almost always tell you exactly what went wrong.

What's Next?

Gameplay Ability System (GAS)

For complex combat, abilities, cooldowns, and status effects, UE5's Gameplay Ability System provides a production-proven framework. It handles ability activation, attribute modification (health, mana, stamina), gameplay effects (buffs, debuffs, damage-over-time), and gameplay tags for state management. GAS is used in Fortnite and many AAA titles.

C++ Integration

When to move from Blueprint to C++:

  • Performance-critical logic: AI decision trees running every frame, complex math, large data processing
  • Base classes: Write core systems in C++, expose them to Blueprint for designers to extend
  • Engine access: Some engine features are only available in C++
  • Team workflows: C++ for programmers, Blueprints for designers — both contributing to the same project
The sweet spot: C++ for the foundation (base character, game mode, core systems). Blueprints for content-specific behaviour (this specific enemy's attack pattern, this level's puzzle logic).

Plugins & Marketplace

  • Fab (formerly Marketplace): Thousands of free and paid assets, plugins, and complete starter projects
  • Common plugins: Enhanced Input System (built-in), Common UI, Online Subsystem for multiplayer
  • Community plugins: Dialogue systems, quest frameworks, procedural generation tools
  • Building your own: Package reusable Blueprints and C++ code as plugins for use across projects

Portfolio Tips

  • Document your Blueprints: Take high-resolution screenshots of clean, well-commented graphs. Employers want to see how you think, not just the final result.
  • Show before/after: Demonstrate iteration — first prototype vs polished version
  • Record gameplay: Short videos showing your systems in action (30-60 seconds each)
  • Explain your decisions: Write brief descriptions of why you chose certain patterns (components vs inheritance, events vs tick, etc.)
  • Version control: Use Git or Perforce. Show commit history to demonstrate consistent progress.
