Foundation
Mixed Reality in UE5

Understand what mixed reality is, how passthrough cameras blend real and virtual worlds, and what hardware makes it possible.

00

What Is Mixed Reality

Mixed Reality (MR) blends virtual content with the real, physical world around you. Unlike VR, which replaces your environment entirely, MR uses passthrough cameras on the headset to show a live video feed of your room — and then composites digital objects on top. You can see your furniture, your walls, your hands, and interact with holograms that appear to exist in your actual space.

The Reality–Virtuality Continuum

ModeWhat You SeeExample
Real WorldOnly physical environmentNormal vision
Augmented RealityReal world + simple overlaysPhone AR, HUD glasses
Mixed RealityReal world + spatially-aware 3D contentQuest 3 passthrough apps
Virtual RealityFully synthetic environmentImmersive VR games
MR is the middle ground: virtual objects respect the physical world. A virtual ball bounces off your real table, a portal opens on your real wall, a creature hides behind your real couch.

Passthrough Technology

Modern headsets use outward-facing cameras to capture your environment in real time. The headset reconstructs a stereoscopic view and displays it on the internal screens. Quest 3 uses high-resolution colour passthrough cameras, making the blend between real and virtual convincing.

  • Stereoscopic passthrough — two camera feeds give depth perception
  • Depth sensor — time-of-flight sensor allows occlusion (real objects can hide virtual ones)
  • Low latency — camera-to-display pipeline must be fast to avoid motion sickness

Spatial computing is the broader term for this paradigm — computing that understands and interacts with your 3D physical space. MR is how users experience it.

Project & Passthrough Configuration

Create an MR-ready UE5 project with the Meta XR plugin and enable passthrough rendering.

01

Project Setup

Start with the VR Template. MR builds on top of VR — you still need a VR Pawn, tracked controllers, and OpenXR. The VR Template gives you all of this out of the box. From there, you add MR-specific features.

  1. Create a new project using the VR Template (UE 5.3+)
  2. Go to Edit → Plugins and search for Meta XR
  3. Enable Meta XR Platform, Meta XR, and Meta XR Passthrough plugins
  4. Restart the editor when prompted
  5. In Project Settings → OpenXR, ensure Meta Quest OpenXR is enabled

Android Manifest Permissions

MR features require explicit permissions in the Android manifest. These are typically set automatically by the Meta XR plugin, but verify them:

  • com.oculus.permission.USE_SCENE — required for Scene API
  • com.oculus.permission.USE_ANCHOR_API — required for spatial anchors
  • Passthrough is enabled via the OpenXR extension, not a manifest permission
Important: MR features only work on-device. You cannot test passthrough or scene understanding in the editor’s VR Preview on PC — you must deploy to the Quest.
02

Passthrough Setup

Passthrough replaces the skybox with a live camera feed. In UE5 with the Meta XR plugin, this is done via the OXRPassthroughLayer component and a transparent background colour.

  1. Open your VR Pawn Blueprint
  2. Add an OXRPassthroughLayer component to the Pawn
  3. In the component’s Details panel, set Placement to Underlay
  4. In Project Settings → Rendering, set the Clear Color to transparent black (0, 0, 0, 0)
  5. Remove or disable any Sky Sphere, Sky Atmosphere, and Directional Light used for sky rendering

Underlay vs Overlay

ModeBehaviourUse Case
UnderlayPassthrough renders behind everything; transparent pixels show the real worldStandard MR — virtual objects float in real space
OverlayPassthrough renders on top of virtual contentSelective passthrough windows, portals to reality
Underlay is like a green screen in reverse: instead of replacing a green background with a virtual one, you replace a virtual background with the real world. Any pixel your scene doesn’t draw shows the camera feed instead.

Transparent Background

For the passthrough underlay to be visible, your scene’s background must be transparent. This means:

  • Set the level’s Post Process Volume to not render any sky
  • Use Mobile HDR rendering (required for Quest standalone)
  • Ensure no opaque full-screen effects cover the background
Milestone: When you deploy to Quest 3, you should see your real room with the VR Pawn’s motion controllers floating in front of you — no virtual skybox, just your actual space.
Spatial Awareness
Understanding & Anchoring to the Real World

Use the Meta Scene API and spatial anchors to make virtual content aware of and attached to physical space.

03

Scene Understanding

Scene Understanding lets your app know about the physical room. The user runs Space Setup on their Quest, scanning walls, floor, ceiling, doors, windows, and furniture. Your app then queries this data via the Meta Scene API to get 3D meshes and semantic labels for every surface.

How Scene Data Works

  1. The user completes Space Setup in the Quest system menu (scan room boundaries and furniture)
  2. The headset stores a Scene Model — a collection of anchors with semantic labels and 2D/3D bounds
  3. Your app requests the scene using the Scene Capture or Scene Discovery flow
  4. You receive room anchors for walls, floor, ceiling, desks, couches, etc.
  5. Each anchor has a position, rotation, bounding box, and semantic label

Using Scene Data in Blueprints

The Meta XR plugin provides Blueprint nodes for scene queries:

  • Request Scene Capture — prompts the user to run Space Setup if no scene exists
  • Get Scene Anchors — returns all anchors matching a semantic label filter
  • Get Anchor Bounds — returns the 2D/3D bounding rectangle of an anchor
  • Get Global Mesh — retrieves the room’s triangle mesh for physics collisions
Tip: Use the global mesh as an invisible collision surface. This lets virtual objects bounce off real walls and roll across real tables without needing to manually place colliders.
04

Spatial Anchors

Spatial Anchors let you pin virtual content to a specific real-world location. When the user returns to the app later, the anchor persists — objects appear in the same physical spot. This is essential for MR experiences that feel permanent and grounded.

Anchor Lifecycle

  1. Create — call Create Spatial Anchor at a world-space position/rotation
  2. Save — persist the anchor to local storage so it survives app restarts
  3. Load — on next launch, query saved anchors and re-attach your content
  4. Erase — delete anchors the user no longer needs

Persistence Modes

ModeScopeUse Case
LocalSaved on this headset onlyPersonal room decorations, saved game state
Cloud (Shared)Shared across headsets via Meta cloudMultiplayer MR, collaborative design
Think of spatial anchors as invisible thumbtacks. You press a thumbtack into a spot in your room, and the virtual poster you hang on it stays there — even when you leave and come back the next day.

Blueprint Workflow

  • Use Create Spatial Anchor at the player’s hand or controller position when they confirm placement
  • Attach your Actor (mesh, widget, effect) to the anchor’s transform
  • Call Save Spatial Anchor to persist; store the returned UUID for later retrieval
  • On app launch, call Query Spatial Anchors to reload saved anchors and re-spawn content
Milestone: Place a virtual object in your room, close and reopen the app, and the object should reappear in exactly the same spot.
Interaction & Design
Hands, Physics, and MR Design Principles

Build natural interactions with hand tracking and physics, then apply design principles that make virtual content feel real.

05

Interaction in MR

Hand tracking is the default in MR. Because users can see their real hands through passthrough, controller-free interaction feels natural. The Meta XR plugin provides full 26-joint hand skeleton tracking with pinch and poke detection.

Interaction Types

GestureDetectionUse Case
PinchThumb + index finger close togetherGrab objects, select UI buttons at a distance
PokeIndex finger extends and touches a surfacePress virtual buttons, tap UI panels
GrabFull hand close around an objectPick up and move virtual items
RayPinch + wrist orientation casts a rayPoint at distant objects or menus

Physics on Real Surfaces

One of MR’s most compelling features is virtual objects interacting with real-world geometry:

  • Use the global mesh from Scene Understanding as a collision surface
  • Apply the mesh as an invisible static mesh with collision — no visible material
  • Virtual objects with Simulate Physics enabled will collide with real walls, bounce off real tables, and roll across your real floor
  • Use Physics Materials to control bounce, friction, and restitution
Imagine dropping a virtual rubber ball in your real kitchen. It bounces off your real countertop, rolls across your real floor, and comes to rest against your real wall. That is MR physics.

Controller Fallback

Always support both hand tracking and controllers. Some users prefer controllers, and tracking can fail in poor lighting. The Meta XR plugin can switch automatically between input modes.

Best practice: Design interactions that work with hands first, then verify they also work with controllers. Pinch maps naturally to the trigger button; poke maps to a forward press.
06

Design Principles

MR design is about believable coexistence. Virtual content must look and behave as though it truly exists in the user’s physical space. This requires careful attention to lighting, depth, performance, and user comfort.

Blending Virtual and Real

  • Match real-world lighting — use environment probes or manual light matching so virtual objects cast believable shadows
  • Use shadows on real surfaces — render virtual shadows onto the scene mesh to ground objects visually
  • Scale matters — virtual objects must be exactly the right size; a coffee cup should look like a real coffee cup
  • Occlusion — use the depth sensor and scene mesh to occlude virtual objects behind real ones

Depth Cues

  • Cast shadows from virtual objects onto real surfaces
  • Use occlusion so real furniture hides virtual objects
  • Add subtle ambient occlusion where virtual objects meet real surfaces
  • Consider edge glow to help virtual objects read against complex backgrounds

Performance

  • Target 72 Hz minimum (90 Hz ideal) on Quest 3
  • Passthrough adds GPU cost — budget fewer draw calls than pure VR
  • Use Forward Rendering and Mobile HDR
  • Keep triangle counts low for virtual objects
  • Profile with OVR Metrics Tool

Comfort Guidelines

PrincipleWhy
Never move the passthrough layerShifting the camera feed causes instant nausea
Keep virtual UI in comfortable range0.5m – 3m from the user; avoid near-eye content
Avoid full-screen effectsFlash, bloom, or post-process that covers passthrough is disorienting
Respect room boundariesVirtual content should not encourage walking into walls
Provide a clear exitUsers must always be able to see enough of their real room to stay safe
Safety first: Unlike VR, MR users are in their real environment. Your app must never obscure hazards, doorways, or other people. Meta’s guidelines recommend keeping at least 20% of the passthrough visible at all times.

✓

Final Checklist

Before you call your MR project complete, verify each element:

  • Meta XR plugins enabled — Passthrough, Scene, Anchors, and Hand Tracking
  • Passthrough rendering with OXRPassthroughLayer underlay and transparent background
  • Scene Understanding querying room anchors (floor, walls, furniture) from the Scene API
  • Global mesh collision enabling virtual physics on real surfaces
  • Spatial Anchors pinning content to real locations with persistence across sessions
  • Hand tracking working with pinch, poke, and grab gestures
  • Controller fallback functional when hand tracking is unavailable
  • Shadows and occlusion making virtual objects look grounded in the real world
  • Performance stable at 72+ Hz on Quest 3 with passthrough active
  • Comfort verified — no passthrough movement, safe boundaries, clear exit
  • On-device testing completed in a real room with Space Setup data
  • Android permissions set for USE_SCENE and USE_ANCHOR_API
