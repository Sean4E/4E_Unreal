Timelines — Smooth Animations

A Timeline node is a mini animation curve editor that lives inside your Blueprint. Instead of jumping instantly from A to B, a Timeline lets you smoothly interpolate values over time — perfect for doors sliding open, platforms moving, lights fading, or any gradual change.

Think of a Timeline like an animation keyframe editor. You define "at time 0 the value is 0, at time 2 the value is 1" and the Timeline smoothly outputs every value in between, every frame.

Creating a Timeline

  1. In the Event Graph, right-click and search for Add Timeline
  2. Name it DoorTimeline
  3. Double-click the Timeline node to open the curve editor
  4. Click + Float Track to add a new float curve
  5. Name the track Alpha
  6. Right-click on the curve → Add Key at time 0.0, value 0.0
  7. Add another key at time 2.0, value 1.0
  8. You now have a smooth 0-to-1 ramp over 2 seconds

Looping: Check the Loop checkbox in the Timeline editor for continuous animations like bobbing platforms or pulsing lights. The Finished pin will never fire on a looping timeline.

Lerp (Linear Interpolate)

The Lerp node blends between two values using an alpha (0 to 1). Feed the Timeline's alpha output into Lerp and you get smooth motion between any two positions, rotations, or colours.

  • A — the starting value (alpha = 0)
  • B — the ending value (alpha = 1)
  • Alpha — how far between A and B (from your Timeline)

Practical: BP_SlidingDoor

A door that slides open smoothly when the player overlaps a trigger, and slides closed when they leave.

  1. Create a new Actor Blueprint called BP_SlidingDoor
  2. Add a Static Mesh (the door panel) and a Box Collision (the trigger zone)
  3. Create a Timeline called DoorTimeline with a float track from 0 to 1 over 2 seconds
  4. On BeginOverlap → call Play from Start on the Timeline
  5. On EndOverlap → call Reverse from End on the Timeline
  6. From Update → wire to a Lerp (Vector) node
  7. A = closed position (0, 0, 0), B = open position (0, 300, 0)
  8. Wire the Lerp output to Set Relative Location on the door mesh

Functions — Reusable Logic

A function is a named group of nodes you can call repeatedly. Instead of duplicating the same cluster of nodes in five places, you put them inside a function and call it by name. Change the function once, and every call updates automatically.

Functions are like recipes. Instead of writing out "crack eggs, add flour, mix, pour, bake" every time someone asks for cake, you write the recipe once and just say "make cake." The recipe is the function. Saying "make cake" is calling it.

Creating a Function

  1. In the My Blueprint panel (bottom-left), find the Functions section
  2. Click the + button next to Functions
  3. Name your function (e.g. ApplyDamage)
  4. A new graph tab opens — this is your function's internal logic
  5. The purple entry node is where execution begins when the function is called
  6. Add nodes, then add a Return Node if you need to output a value

Inputs and Outputs

Functions can accept parameters (inputs) and return values (outputs):

  • Select the function's entry node and look in the Details Panel
  • Under Inputs, click + to add parameters. Each has a name and type.
  • Under Outputs, click + to add return values.
Tip: When you add an output, a Return Node automatically appears in the function graph. Wire your result into it.

Pure Functions

Pure functions have no execution (white) pins. They simply compute a value and return it. Pure functions are called automatically whenever something needs their output — you do not need to wire an exec pin.

  • Check Pure in the function's Details Panel to make it pure
  • The node turns green in the graph
  • Best for: calculations, getters, checks (e.g. GetHealthPercent, IsAlive)
Warning: Pure functions are called every time their output is read. Do not put expensive logic (traces, loops, spawns) inside a pure function — it may execute multiple times per frame.

Local Variables

Functions can have their own local variables that exist only while the function runs. They do not pollute your main variable list.

  • In the function graph, look in the My Blueprint panel under Local Variables
  • Click + to add a local variable
  • These are temporary — they reset every time the function is called

Custom Events & Event Dispatchers

Custom Events are your own named events that you can call from anywhere in the same Blueprint. Unlike functions, they can contain latent nodes like Delay and they fire asynchronously — execution continues immediately after calling them.

  1. Right-click in the Event Graph → Add Custom Event
  2. Name it (e.g. OpenDoor)
  3. Build your logic chain from the event node
  4. To trigger it, right-click → search Call OpenDoor (or drag from an exec pin)

Event Dispatchers — Broadcasting

Event Dispatchers let one Blueprint broadcast a signal that other Blueprints can listen to. This is the observer pattern — "when something happens to me, I will let everyone who is listening know."

Event Dispatchers are like a radio station. The station broadcasts a signal (dispatches). Anyone who tunes in (binds) hears the broadcast. The station does not need to know who is listening.

Creating and Using Dispatchers

  1. In My Blueprint panel, find Event Dispatchers → click +
  2. Name it OnPressed
  3. Optionally add parameters (inputs) in the Details Panel
  4. In the owning Blueprint, drag OnPressed into the graph → choose Call
  5. In the listening Blueprint, get a reference to the owning actor
  6. Drag OnPressed → choose Bind
  7. Wire the Event output to a Custom Event that handles the response

Unbinding and Cleanup

  • Unbind Event — stops a specific listener from receiving the dispatch
  • Unbind All Events — removes all listeners at once
  • Always unbind in Event EndPlay to prevent dangling references when actors are destroyed
Memory safety: If you destroy an actor that is still bound to a dispatcher, you may get errors. Always unbind before destroying or use weak references.
Milestone: You can create Custom Events, set up Event Dispatchers for cross-Blueprint communication, bind and unbind listeners, and understand the observer pattern in Blueprints.

Arrays & Loops

An Array is an ordered list of items that all share the same type. Need to store 10 waypoints? An array of Vectors. A list of inventory items? An array of Structs. Arrays are the most common collection type in Blueprints.

An array is like a numbered shelf. Slot 0 holds the first item, slot 1 holds the second, and so on. You can add items, remove items, or ask "what is in slot 3?"

Creating an Array Variable

  1. In the My Blueprint panel, click + under Variables
  2. Name it Waypoints
  3. Set the type to Vector
  4. Click the icon next to the type and select the Array grid icon
  5. The variable icon changes to show a grid — it is now an array of Vectors

Essential Array Nodes

NodeWhat It Does

ForEachLoop

Iterates over every item in the array, executing the Loop Body once per item. The Array Element output gives you the current item, and Array Index tells you which slot you are on.

Completed: The Completed exec pin fires after all items have been processed.

ForEachLoopWithBreak

Same as ForEachLoop, but with an extra Break exec input. Wire a condition to Break and the loop stops early — useful for "find the first enemy in range" patterns.

Practical: BP_WaypointPatrol

An actor that moves between an array of world positions in sequence.

  1. Create BP_WaypointPatrol with a Static Mesh and a variable Waypoints (Array of Vector, Editable)
  2. Add an Integer variable CurrentIndex (default 0)
  3. On BeginPlay, call a Custom Event MoveToNextWaypoint
  4. In MoveToNextWaypoint: Get from Waypoints at CurrentIndex
  5. Use a Timeline + Lerp (from Step 8) to move the actor to that position
  6. On Timeline Finished: increment CurrentIndex, wrap using % (modulo) Length
  7. Call MoveToNextWaypoint again — the patrol loops forever

Structs & Data Tables

A Struct is a custom data type that bundles multiple variables together. Instead of storing a weapon's name, damage, fire rate, and icon as four separate variables, you create one Struct type that holds all four. Then each weapon is a single variable of that Struct type.

A Struct is like a form template. The template says "every entry must have a Name, Damage, Fire Rate, and Icon." Each filled-in form is an instance of that Struct.

Creating a Struct

  1. In the Content Browser, right-click → Blueprints → Structure
  2. Name it S_WeaponData
  3. Double-click to open the Struct editor
  4. Click + Add Variable for each field:
⚙ S_WeaponData — Structure Editor
Variables
WeaponNameString

Break Struct / Make Struct

Two essential nodes for working with Structs:

  • Break S_WeaponData — takes a struct and splits it into its individual fields (one output pin per variable)
  • Make S_WeaponData — takes individual values and combines them into a struct (one input pin per variable)
Shortcut: Drag from a struct pin and choose Break or Make from the context menu to quickly create these nodes.

Data Tables

A Data Table is a spreadsheet-style asset where each row is an instance of a Struct. Perfect for game data that designers need to tweak without opening Blueprints: weapon stats, enemy configurations, item databases, dialogue lines.

  1. In the Content Browser, right-click → Miscellaneous → Data Table
  2. Select S_WeaponData as the row structure
  3. Name it DT_Weapons
  4. Double-click to open the Data Table editor
  5. Click + Add to create rows — each row has a unique Row Name (the key)
  6. Fill in the fields for each weapon

Interfaces — Clean Communication

A Blueprint Interface is a contract that says "I can do X." Any Blueprint that implements the interface promises to handle its functions. The caller does not need to know what type of actor it is talking to — only that the actor implements the interface.

An interface is like a power outlet standard. Your phone charger does not care if the wall outlet is in your kitchen or your bedroom. It just needs a standard plug shape (the interface). Any room that implements that shape can charge your phone.

Creating a Blueprint Interface

  1. In the Content Browser, right-click → Blueprints → Blueprint Interface
  2. Name it BPI_Interactable
  3. Double-click to open — you see a function list
  4. Rename the default function to Interact
  5. Add an input: Caller of type Actor
  6. Save and close
◆ BPI_Interactable — Interface Functions
Functions
InteractCaller: Actor

Implementing an Interface

  1. Open any Blueprint (e.g. BP_Door)
  2. Go to Class Settings (top toolbar)
  3. In the Details Panel, find Interfaces → click Add
  4. Select BPI_Interactable
  5. Compile — the function Interact now appears under Interfaces in My Blueprint
  6. Double-click Interact to open its graph and add your door-opening logic
Each BP implements differently: BP_Door opens the door. BP_Light toggles visibility. BP_Pickup collects the item. All through the same Interact function.

Calling Through an Interface

Two ways to call an interface function:

MethodHowWhen
MessageRight-click → search "Interact (Message)". If the target does not implement the interface, nothing happens (safe).Default. Use this most of the time.
Direct CallCast to the interface, then call Interact directly. Crashes if the target does not implement it.Only when you are certain the target implements it.

Does Implement Interface

Use the Does Implement Interface node to check before calling:

  • Input: an Actor reference
  • Output: Boolean — true if the actor implements the interface
  • Wire into a Branch to safely guard the call

Practical: Universal Interaction System

A single interaction flow that works on any interactable object:

  1. Player presses E
  2. Fire a Line Trace from the camera forward
  3. If the trace hits something, call Interact (Message) on the hit actor, passing Self as Caller
  4. If the actor is a Door, it opens. If it is a Light, it toggles. If it is a Pickup, it collects. The player does not need to know.

Widget Blueprints — Basic HUD

Widget Blueprints (WBP) are UE5's system for creating 2D user interface elements — health bars, score displays, menus, inventory screens, and crosshairs. They use a visual designer similar to web layout tools, with drag-and-drop widgets on a canvas.

Creating a Widget Blueprint

  1. In the Content Browser, right-click → User Interface → Widget Blueprint
  2. Name it WBP_HUD
  3. Double-click to open the Widget Designer
  4. You see two tabs: Designer (visual layout) and Graph (Blueprint logic)

Key Widget Types

WidgetPurpose
Canvas PanelThe root container. All other widgets go inside it. Allows absolute positioning.
TextDisplay text on screen (labels, scores, messages)
Progress BarA fill bar — perfect for health, mana, loading
ImageDisplay a texture or icon
Horizontal/Vertical BoxLayout containers that arrange children in a row or column
ButtonClickable element for menus

Anchors

Anchors pin a widget to a specific part of the screen. When the screen resizes, the widget stays relative to its anchor point.

  • Top-Left — default, widget stays near top-left corner
  • Top-Centre — score display that stays centred at the top
  • Bottom-Left — health bar in the lower-left
  • Stretch — widget fills the entire screen (useful for overlays)
Tip: Select a widget and click the anchor preset dropdown (the flower-like icon) in the Details Panel to quickly set common anchor positions.

Bindings — Dynamic Data

Bindings link widget properties to game variables so the UI updates automatically:

  1. Select the Progress Bar in the Designer
  2. In the Details Panel, find Percent
  3. Click Bind → Create Binding
  4. A new function graph opens — this runs every frame
  5. Inside it, get a reference to the player, read their Health, divide by MaxHealth
  6. Wire the result (0.0 to 1.0) to the Return Value
Performance note: Bindings run every frame. For complex UI, consider using Event-Driven updates instead — only update the widget when the value actually changes.

Displaying the HUD

Widgets must be created and added to the viewport at runtime:

  1. Open your PlayerController Blueprint (or create one)
  2. On BeginPlay, add a Create Widget node
  3. Set the Class to WBP_HUD
  4. Wire the output to Add to Viewport
  5. Optionally, store the widget reference in a variable for later access

Practical: Health Bar

A progress bar that decreases when the player takes damage:

  1. In your Player Blueprint, add variables: Health (Float, default 100) and MaxHealth (Float, default 100)
  2. In WBP_HUD, bind the Progress Bar's Percent to Health / MaxHealth

AI Basics — Simple Enemy Behaviour

UE5's AI framework gives NPCs the ability to perceive their environment, make decisions, and navigate the world. At the simplest level, you need three things: an AI Controller (the brain), a Pawn (the body), and a Nav Mesh (the walkable area).

Think of it like a remote-controlled car. The AI Controller is the person holding the remote. The Pawn is the car itself. The Nav Mesh is the map of roads the car is allowed to drive on.

Core AI Components

ComponentWhat It Does
AIControllerThe "brain" that possesses and controls a Pawn. Contains decision-making logic.
Pawn / CharacterThe physical body the AI controls. Has a mesh, collision, and movement component.
NavMeshAn invisible mesh baked over your level geometry that tells AI where it can walk.
Behavior Tree(Advanced) A tree of tasks and decisions. The industry standard for complex AI.
Blackboard(Advanced) A key-value store that the Behavior Tree reads. Stores data like "last known player location."
For this step, we will build simple AI using Blueprints only — no Behavior Trees. BTs are powerful but deserve their own lesson.

Setting Up the NavMesh

  1. In your level, go to the Place Actors panel (Shift+1)
  2. Search for NavMeshBoundsVolume
  3. Drag it into the level and scale it to cover the entire walkable area
  4. Press P to toggle NavMesh visualisation — green overlay shows walkable areas
  5. The NavMesh automatically avoids walls, obstacles, and edges
No green? Make sure your floor has collision enabled and the NavMeshBoundsVolume fully covers the area. Rebuild paths if needed (Build menu → Build Paths).

Simple Patrol (No Behavior Tree)

Using the waypoint array from Step 11, we can make an enemy patrol between points:

  1. Create BP_Enemy as a Character Blueprint (has built-in movement)
  2. Add a Sphere Collision component (detection radius, e.g. 1000 units)
  3. Add variables: Waypoints (Array of Vector), CurrentIndex (Integer), bPlayerDetected (Boolean)
  4. On BeginPlay → call AI MoveTo with the first waypoint
  5. When AI MoveTo completes (On Success) → increment index, call AI MoveTo with next waypoint
  6. Wrap the index with modulo so the patrol loops

Detect the Player

Use the Sphere Collision component as a detection zone:

  • On Begin Overlap → check if Other Actor is the Player (Cast to your Character class)
  • If yes → set bPlayerDetected = true
  • On End Overlap → set bPlayerDetected = false

AI MoveTo Node

The AI MoveTo node tells an AI-controlled pawn to navigate to a location using the NavMesh:

  • Target Actor or Destination — where to go
  • Acceptance Radius — how close is "close enough" (e.g. 50 units)
  • On Success — exec fires when the AI reaches the destination
  • On Fail — exec fires if the AI cannot find a path
Requires NavMesh: AI MoveTo will fail silently if there is no NavMeshBoundsVolume in your level. Always set up the NavMesh first.

Practical: Patrol + Chase Enemy

Putting it all together — an enemy that patrols, chases the player when near, and returns to patrol when the player escapes:

  1. On BeginPlay, start patrolling (AI MoveTo first waypoint)
  2. On Sphere Begin Overlap (player enters), set bPlayerDetected = true
  3. Use a Timer by Event (every 0.5s) to check: if bPlayerDetected, call AI MoveTo targeting the player
  4. On Sphere End Overlap (player leaves), set bPlayerDetected = false
  5. When the timer fires and bPlayerDetected is false, resume patrol — AI MoveTo the next waypoint
  6. Adjust the Sphere Collision radius to control detection range
