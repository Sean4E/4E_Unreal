AR in UE5

Why Unreal Engine for AR? UE5 provides native support for ARKit (iOS) and ARCore (Android) through its AR subsystem. The engine handles camera passthrough, motion tracking, plane detection, and light estimation — letting you focus on the creative experience rather than low-level device APIs.

ARKit vs ARCore

FeatureARKit (iOS)ARCore (Android)
Plane DetectionHorizontal + VerticalHorizontal + Vertical
Light EstimationAmbient intensity + color temperatureAmbient intensity + color correction
Image TrackingUp to 100 reference imagesUp to 20 reference images
Face TrackingTrueDepth camera (front)Front camera (limited)
OcclusionLiDAR-based (Pro/iPad Pro)Depth API (select devices)
Think of ARKit and ARCore as two dialects of the same language. UE5’s AR subsystem speaks both — you write your logic once and it works on either platform with minimal changes.

What We’ll Build

A handheld AR application that detects real-world surfaces through the phone camera, lets the user tap to place 3D objects on those surfaces, and uses light estimation to blend virtual content seamlessly with the real environment.

  • Surface detection — visualise detected planes in real time
  • Object placement — tap to spawn actors aligned to surfaces
  • Realistic lighting — match virtual light to real-world conditions
  • Mobile deployment — package and run on iOS or Android
Scope: This guide covers handheld AR (phone/tablet). Head-mounted AR (HoloLens, Magic Leap) uses a different UE5 workflow via OpenXR.

Project Setup

Create the Project

  1. Open UE5 and select Games → Handheld AR template
  2. Choose Blueprint as the project type
  3. Set target platform to Mobile (scalable)
  4. Name the project (e.g. AR_PlaceObjects) and create
Why this template? The Handheld AR template ships with AR Session Config, a basic AR Pawn, and touch input — saving significant setup time.

Enable Required Plugins

Go to Edit → Plugins and confirm these are enabled:

AR Session & Tracking

AR Session Config

The AR Session Config data asset controls what AR features are active. It is the central configuration for your entire AR experience.

  1. In Content Browser, right-click → Miscellaneous → Data Asset → ARSessionConfig
  2. Open it and configure the settings (see Details Panel below)
  3. In your AR Pawn or GameMode, call Start AR Session passing this config

03

Surface Detection

How Plane Detection Works

Once the AR Session is running with plane detection enabled, the device scans the environment and reports AR Tracked Geometries (planes). Each plane has a position, rotation, extent, and classification (floor, wall, table, ceiling).

  • Horizontal planes — floors, tables, desks, shelves
  • Vertical planes — walls, doors, cabinets

Visualising Detected Planes

The Handheld AR template includes a plane visualisation Blueprint. Key nodes:

// On each Tick or AR event
Get All AR Geometries → For Each Loop
  → Get Object Classification
  → Get Local to World Transform
  → Spawn / Update PlaneVisualizerActor
Tip: Use a semi-transparent material on plane visualisers so users can see the real world through them. A subtle grid or pulse effect helps communicate “surface detected”.

Filtering Planes

Not every detected plane is useful. Filter by classification:

ClassificationUse Case
FloorPlacing furniture, characters, large objects
TablePlacing small props, tabletop games
WallHanging art, mounting screens
None / UnknownOften noisy — consider ignoring

04

Place Objects

Touch-to-Place Workflow

The core AR interaction: the user taps the screen, a line trace fires from the touch point into the AR world, hits a detected plane, and spawns an actor at that location.

  1. Capture touch input with Input Touch event
  2. Convert 2D screen position to a 3D world ray with Deproject Screen to World
  3. Perform Line Trace by Channel against AR geometry
  4. If hit, Spawn Actor at the hit location and align rotation to the surface normal

Blueprint: Line Trace from Touch

// Input Touch → Pressed
Get Touch Location (Finger Index 0)
  → Deproject Screen to World
    → World Position, World Direction

Line Trace by Channel
  Start: World Position
  End: World Position + (World Direction × 5000)
  → Hit Result

Branch: Hit?
  True → Spawn Actor from Class
    Class: BP_ARPlaceable
    Location: Hit Location
    Rotation: Make Rot from Z(Hit Normal)

Aligning to Surface

Use the hit normal to orient the spawned object. For a floor placement, the normal is (0, 0, 1) so the object stands upright. For a wall, the normal faces outward and the object mounts flat.

Scale matters. AR objects use real-world units (centimetres). A 100-unit cube in UE5 = 1 metre. If your object appears gigantic, scale it down — most AR props should be 10–50cm.

AR Lighting & Realism

Light Estimation

AR frameworks analyse the camera feed to estimate ambient light. UE5 exposes this through the AR Light Estimate node, which returns:

  • Ambient Intensity — overall brightness (0–2000+ lumens)
  • Ambient Color Temperature — warm (low K) to cool (high K)
  • Color Correction — RGBA tint to match the scene (ARCore)

Applying Light Data

// On Tick in AR Pawn
Get AR Light Estimate
  → Ambient Intensity
  → Ambient Color

// Update a Directional Light in the scene
Set Intensity(DirectionalLight, AmbientIntensity × 0.5)
Set Light Color(DirectionalLight, AmbientColor)
Light estimation is like giving your virtual objects “eyes” — they can now see how bright and what colour the real room is, and dress to match.

Shadows & Grounding

Without shadows, objects float. Add a shadow-catcher plane beneath placed objects:

  • Create a flat plane mesh slightly larger than the object footprint
  • Apply a shadow-only material (Blend Mode: Translucent, receives shadows but is otherwise invisible)
  • Parent it to the placed actor so it moves with the object
Shadow material: In the material, set Opacity = 0 but enable Receive Dynamic Shadows. Use a dark translucent colour multiplied by shadow attenuation.

Occlusion

People occlusion (ARKit 3+) and depth occlusion allow real-world objects to appear in front of virtual ones, preventing the “X-ray” effect.

Deploy to Device

iOS Deployment

  1. Install Xcode on a Mac (required for iOS builds)
  2. Set up an Apple Developer account and create a provisioning profile
  3. In Project Settings → iOS, set the Bundle Identifier (e.g. com.studio.arproject)
  4. Set Minimum iOS Version to 14.0 (ARKit 4+ support)
  5. Add a Camera Usage Description (required by Apple for AR apps)
  6. Connect your iPhone/iPad via USB or use Remote Build from a Windows machine via a Mac on the network
  7. Go to Platforms → iOS → Launch
macOS required. iOS packaging needs Xcode, which only runs on macOS. Use a Mac, a VM, or the UE5 Remote Build feature to compile from Windows via a networked Mac.

Android Deployment

  1. Install Android Studio and the Android SDK/NDK (or use UE5’s built-in SDK manager)
  2. In Project Settings → Android, set the Package Name (e.g. com.studio.arproject)
  3. Set Minimum SDK to 24 and Target SDK to 33
  4. Enable Developer Mode and USB Debugging on the Android device
  5. Connect via USB — accept the debugging prompt on the phone
  6. Go to Platforms → Android → Launch
  7. First build takes 10–20 minutes; subsequent builds are faster

✓

Final Checklist

Before you call your AR project complete, verify each element:

  • ARKit and ARCore plugins enabled with the Augmented Reality framework plugin active
  • AR Session Config created with plane detection, light estimation, and auto focus enabled
  • AR Session starts in BeginPlay and tracking quality is monitored
  • Plane detection working — horizontal and vertical surfaces are found and visualised
  • Plane filtering applied — only relevant surface types are shown to the user
  • Touch-to-place functional — line trace from screen touch spawns actors on surfaces
  • Object alignment correct — spawned actors orient to the surface normal
  • Scale is realistic — objects appear at real-world proportions (cm units)
  • Light estimation drives a directional light to match ambient conditions
  • Shadow catcher grounds objects with contact shadows on the real surface
  • Camera permission set in project settings for both iOS and Android
  • Deployed and tested on a physical device (not just the editor)
