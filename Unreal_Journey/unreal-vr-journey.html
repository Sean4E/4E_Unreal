00

VR in UE5

Why Unreal Engine for VR? UE5 provides a production-grade VR pipeline out of the box. The engine ships with an OpenXR integration, a ready-made VR Template project, and deep support for motion controllers, hand tracking, and spatial audio. Whether you are building an architectural walkthrough or an interactive game, UE5 handles the low-level headset communication so you can focus on the experience.

OpenXR — The Universal Standard

OpenXR is an open API standard managed by the Khronos Group. It provides a single interface between your application and any compliant VR headset. UE5 uses OpenXR as its primary VR runtime.

ConceptWhat It Means
OpenXR RuntimeSoftware layer on the PC that translates headset data into a common API
Action SystemInput abstraction — you bind actions (grab, teleport) not buttons
Extension PluginsHeadset-specific features (hand tracking, passthrough) exposed via extensions
Think of OpenXR as a universal translator. Your UE5 project speaks one language, and OpenXR translates it into whatever dialect the connected headset understands — Quest, Vive, Index, or Windows MR.

Supported Headsets

  • Meta Quest 2 / 3 / Pro — standalone (Android) or PC VR via Link/Air Link
  • Valve Index — PC VR via SteamVR OpenXR runtime
  • HTC Vive / Vive Pro / XR Elite — PC VR or standalone (XR Elite)
  • Windows Mixed Reality — HP Reverb G2 and others via WMR OpenXR runtime
  • Apple Vision Pro — via visionOS plugin (separate workflow)

Two Development Approaches

ApproachBest ForHow It Works
VR TemplateRapid prototyping, learningPre-built VR Pawn with teleportation, grab, and menu — start immediately
Custom from BlankFull control, production projectsBuild your own VR Pawn, input mappings, and interaction from scratch
Recommendation: Start with the VR Template. Study how it works, then customise or rebuild components as you gain confidence. The Template is well-documented and follows Epic’s best practices.

01

Project Setup

A properly configured project is essential for VR. Frame rate, rendering settings, and plugins all need to be set before you start building. The VR Template handles most of this automatically.

Create a VR Template Project

  1. Open the Epic Games Launcher → Launch Unreal Engine 5.4+
  2. Select Games → choose the VR template
  3. Set project type to Blueprint (or C++ if preferred)
  4. Choose a project name and location → click Create
  5. The template opens with a pre-built VR level including teleportation, grab objects, and a menu

Essential Plugins

Go to Edit → Plugins and verify these are enabled:

PluginPurposeStatus
OpenXRCore VR runtime communicationEnabled by default
OpenXR Hand TrackingSkeletal hand tracking (Quest, Index)Enable manually
Enhanced InputModern input mapping systemEnabled by default
XR Interaction ToolkitGrab, teleport, UI interaction helpersEnable if available
Restart required: After enabling or disabling plugins, UE5 will prompt you to restart the editor. Save your work first.

Critical Project Settings

Navigate to Edit → Project Settings and configure:

  1. Engine → Rendering → Forward Shading — enable for better VR performance (or use Deferred if you need Lumen)
  2. Engine → Rendering → Instanced Stereo — enable to render both eyes in a single pass
  3. Engine → Rendering → Mobile Multi-View — enable for Quest standalone builds
  4. Project → Target Hardware — set to Mobile / Scalable for Quest, or Desktop / Maximum for PC VR
  5. Platforms → Android — configure if targeting Quest standalone (Package Name, SDK paths)

VR Pawn

The VR Pawn is the player in VR — it represents the user’s body, head, and hands in the virtual world. It consists of a Camera Component (the headset), Motion Controller Components (left and right hands), and a VR Origin that anchors everything to the play space.

Building a VR Pawn from Scratch

  1. Create a new Blueprint Class based on Pawn — name it BP_VRPawn
  2. Add a Scene Component as the root — rename to VROrigin
  3. Add a Camera Component as a child of VROrigin — leave at origin (0, 0, 0)
  4. Add two Motion Controller Components as children of VROrigin
  5. Set one to Left and the other to Right in the Details Panel → Motion Source
  6. Attach hand meshes (skeletal or static) to each Motion Controller
  7. In your Game Mode, set the Default Pawn Class to BP_VRPawn

VR Origin — Why It Matters

The VROrigin component is the anchor point for all tracking. The headset and controllers report their positions relative to this origin. When you teleport the player, you move the VROrigin — not the Camera. The Camera’s position is always controlled by the headset hardware.

Key rule: Never set the Camera’s position or rotation in Blueprint. The headset tracking overwrites it every frame. Move the VROrigin instead and the Camera will follow.
Milestone: You have a VR Pawn with a tracked camera and two motion controllers. When you put on the headset, you can see your hands and look around the scene.

03

Teleportation

Teleportation is the most common VR locomotion method because it avoids motion sickness. The player aims an arc with a controller, sees a destination marker, and presses a button to instantly move there. UE5 uses the Navigation System to determine valid landing spots.

Set Up the Nav Mesh

  1. Place a Nav Mesh Bounds Volume in your level (Place Actors → search “Nav Mesh”)
  2. Scale it to cover the entire walkable area of your environment
  3. Press P to toggle Nav Mesh visualisation — green areas are walkable
  4. Adjust the volume’s size and position until all intended walkable surfaces are covered
Tip: The Nav Mesh only generates on flat or gently sloped surfaces. Steep walls and cliffs are automatically excluded. You can adjust slope tolerance in Project Settings → Navigation Mesh.

Teleport Arc Trace

The teleportation system uses a projectile path prediction to draw a curved arc from the controller:

  1. On thumbstick forward, call Predict Projectile Path By TraceChannel
  2. Set the Start Location to the right Motion Controller’s world position
  3. Set the Launch Velocity to the controller’s forward vector × a speed value (e.g. 900)
  4. Draw the arc using a Spline Component updated each frame
  5. Check if the hit location falls on the Nav Mesh using Project Point to Navigation
  6. If valid, show a destination marker (a ring or disc decal) at the projected point

Execute the Teleport

  1. On thumbstick release, check if the destination is valid
  2. Calculate the offset: Destination - Camera World Position + VROrigin Position
  3. Set the VROrigin’s world location to this offset (keeps the player’s head at the aimed point)
  4. Optionally fade to black and back for a smoother transition
The offset calculation matters because the Camera is not at the VROrigin — the user has physically moved their head. You need to account for that offset so the player lands exactly where they aimed, not slightly off to the side.

04

Hand Interactions

Picking up, throwing, and manipulating objects is the core of VR interactivity. UE5 uses a combination of physics simulation, overlap detection, and the Enhanced Input system to handle hand-object interactions.

Grab System Overview

ConceptHow It Works
Grab SphereA Sphere Collision component on each hand that detects overlapping objects
Grabbable InterfaceAn interface (BPI_Grabbable) that marks actors as grabbable
Attach to ComponentWhen grip is pressed, the overlapping actor is attached to the Motion Controller
Release & ThrowOn grip release, detach and apply the controller’s velocity for realistic throwing

Building the Grab System

  1. Add a Sphere Collision component to each Motion Controller (radius ~12 cm)
  2. Set its collision to Overlap All Dynamic
  3. Create a Blueprint Interface called BPI_Grabbable with functions: Grab and Release
  4. On Grip press, get overlapping actors → check if they implement BPI_Grabbable
  5. Call Attach Actor to Component with the Motion Controller as the parent
  6. On Grip release, detach and call Set Physics Linear Velocity using the controller’s velocity

Enhanced Input for VR

The Enhanced Input system replaces the legacy input system. It uses Input Actions (abstract events) and Input Mapping Contexts (bindings to physical buttons):

  1. Create Input Actions: IA_Grab_L, IA_Grab_R, IA_Teleport, IA_Menu
  2. Create an Input Mapping Context: IMC_VRDefault
  3. Map IA_Grab_L to Oculus Touch (L) Grip Axis with a Dead Zone modifier
  4. Map IA_Grab_R to Oculus Touch (R) Grip Axis similarly
  5. In BeginPlay, call Add Mapping Context to register IMC_VRDefault
Why Enhanced Input? It decouples actions from specific hardware. The same IA_Grab action works on Quest Touch, Index Knuckles, or Vive Wands — you just add different bindings in the Mapping Context.

Making an Object Grabbable

  1. Create a new Actor Blueprint — e.g. BP_VR_Cube
  2. Add a Static Mesh Component (cube, sphere, or custom mesh)
  3. Enable Simulate Physics on the Static Mesh
  4. In Class Settings, add the BPI_Grabbable interface
  5. Implement the Grab function: disable physics, attach to hand
  6. Implement the Release function: detach, re-enable physics, apply throw velocity

VR UI

Traditional screen-space UI does not work in VR. Instead, you place Widget Components in 3D world space — attached to the player’s hand, floating in the environment, or mounted on surfaces. The player interacts with these using a Widget Interaction Component that acts as a laser pointer.

Create a VR Menu Widget

  1. Right-click in Content Browser → User Interface → Widget Blueprint — name it WBP_VRMenu
  2. Design the menu using standard UMG widgets: buttons, text, sliders, images
  3. Keep the canvas size small — e.g. 400 × 600 pixels
  4. Use large font sizes (18pt+) and large button hit areas — precision is lower in VR

Display the Widget in 3D

  1. In your VR Pawn (or a separate actor), add a Widget Component
  2. Set the Widget Class to WBP_VRMenu
  3. Set Draw Size to match your widget canvas (400 × 600)
  4. Set Space to World (not Screen)
  5. Attach it to the left Motion Controller (wrist-mounted menu) or place it in the scene
  6. Adjust the Scale — typically 0.1, 0.1, 0.1 to make the widget a comfortable size

Widget Interaction (Laser Pointer)

  1. Add a Widget Interaction Component to the right Motion Controller
  2. Set Interaction Distance to 500 (how far the laser reaches)
  3. The component automatically performs a line trace and hovers/clicks on Widget Components
  4. On trigger press, call Widget Interaction → Press Pointer Key (Left Mouse Button)
  5. On trigger release, call Release Pointer Key
  6. Optionally, draw the laser using a Beam Particle or Spline Mesh from the controller to the hit point

06

Comfort & Performance

VR comfort is non-negotiable. Dropped frames, latency, or unexpected camera movement causes motion sickness. Performance optimisation in VR is not optional polish — it is a core requirement.

Frame Rate Targets

HeadsetTarget FPSFrame Budget
Meta Quest 272 Hz (or 90 Hz)13.8 ms (or 11.1 ms)
Meta Quest 390 Hz (or 120 Hz)11.1 ms (or 8.3 ms)
Valve Index90 / 120 / 144 Hz11.1 / 8.3 / 6.9 ms
General RuleNever below 72 HzEvery frame must render in budget
Critical: Unlike flat-screen games, you cannot drop below target FPS “sometimes.” Even occasional frame drops cause nausea. Profile early and often using stat fps, stat unit, and stat gpu.

Performance Optimisation Checklist

  • Draw Calls — merge static meshes, use instanced rendering, reduce unique materials
  • Polycount — use LODs (Level of Detail) on all meshes; keep total scene under 500k tris for Quest
  • Lighting — bake static lighting where possible; avoid dynamic shadows on Quest standalone
  • Textures — use compressed formats (ASTC for Android); limit to 1K or 2K max for most assets
  • Translucency — minimise transparent materials; they are extremely expensive in stereo rendering
  • Post Processing — disable expensive effects: SSR, SSAO, motion blur, depth of field
  • Occlusion Culling — use Precomputed Visibility Volumes to skip rendering hidden objects

Comfort Features

FeatureWhat It DoesImplementation
TeleportationEliminates smooth locomotion sicknessAlready built in Step 03
Snap TurningRotates the player in fixed increments (30° or 45°)On thumbstick left/right, rotate VROrigin by fixed degrees

Snap Turning Implementation

  1. Create an Input Action IA_SnapTurn mapped to right thumbstick X-axis
  2. Add a Dead Zone modifier (threshold ~0.7) so it only fires on deliberate pushes
  3. On Triggered, rotate the VROrigin actor by ±45 degrees around the Camera’s world position
  4. Add a short cooldown (0.3 s) to prevent multiple snaps per push
  5. Optionally, briefly flash a dark screen or play a subtle audio cue on each snap

Vignette Effect

A locomotion vignette reduces the visible field of view during movement, which greatly reduces nausea for sensitive users:

  1. Create a Post Process Material that darkens pixels outside a circular radius
  2. Expose the radius as a scalar parameter
  3. In Blueprint, lerp the radius from 1.0 (full view) to 0.5 (vignetted) based on player velocity
  4. Apply the material via a Post Process Component on the VR Pawn
Milestone: Your project runs at a stable frame rate on the target headset. Snap turning and vignette are available as comfort options.
07

Testing & Deployment

Testing VR requires actually wearing the headset. Desktop previews help with logic and layout, but comfort, scale, and interaction feel can only be evaluated in-headset. UE5 provides multiple preview modes and a full deployment pipeline for standalone headsets.

VR Preview Mode

  1. Connect your headset to the PC (Link cable, Air Link, or SteamVR)
  2. In the UE5 toolbar, click the dropdown next to Play → select VR Preview
  3. The editor launches the level on your headset — put it on and test
  4. Use stat fps and stat unit console commands to monitor performance
  5. Press Esc or the Stop button to exit VR Preview
Tip: VR Preview uses the PC’s GPU, so performance will differ from a standalone Quest build. Always do a final test on the target hardware.

Testing Checklist

  • Scale — do objects feel the right size? Is the player the right height?
  • Comfort — does movement cause any nausea? Test with multiple people
  • Interaction reach — can the player reach buttons and grab objects comfortably?
  • UI readability — is text large enough to read at arm’s length?
  • Performance — is the frame rate stable with no hitches or stutters?
  • Audio — does spatial audio feel correct? Do sounds come from the right directions?

Quest Standalone Deployment

Deploying to Meta Quest without a PC tether requires an Android build:

  1. Install Android Studio and the Android SDK/NDK (or use the UE5 SDK installer in Project Settings)
  2. In Project Settings → Android, set the Package Name (e.g. com.yourname.vrproject)
  3. Set Minimum SDK to 29 and Target SDK to 32
  4. Enable Developer Mode on the Quest via the Meta app on your phone
  5. Connect the Quest via USB — accept the “Allow USB debugging” prompt on the headset
  6. In UE5, go to Platforms → Android → Launch (or use the Launch button with the Quest selected)
  7. Wait for the build, cook, and deploy process to complete (first build takes 10–30 minutes)
  8. The app appears in the Quest under Library → Unknown Sources

PC VR Packaging

For PC VR distribution (SteamVR, Rift):

  1. Go to Platforms → Windows → Package Project
  2. Choose a build configuration: Shipping for release, Development for testing
  3. Select an output folder and wait for the cook and package process
  4. The output is a standalone .exe that launches directly into VR when a headset is connected

✓

Final Checklist

Before you call your VR project complete, verify each element:

  • OpenXR plugin enabled and project configured for your target headset
  • VR Pawn with tracked Camera, left and right Motion Controllers, and hand visuals
  • Teleportation working with arc trace, Nav Mesh validation, and destination marker
  • Grab system functional — objects can be picked up, moved, and thrown with physics
  • Enhanced Input configured with Input Actions and Mapping Context for VR controllers
  • VR UI displayed as a Widget Component in world space with Widget Interaction laser pointer
  • Snap turning implemented with configurable angle (30° or 45°)
  • Comfort vignette available as a toggle for sensitive users
  • Frame rate stable at target Hz on the intended headset (72/90/120 Hz)
  • Performance profiling done with stat fps, stat unit, stat gpu
  • In-headset testing completed for scale, comfort, interaction, and readability
  • Deployment tested — Quest APK or packaged PC VR build runs standalone
